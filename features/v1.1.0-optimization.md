# v1.1.0 优化计划

## 认证改进

### Authing 用户绑定

**问题**: 当前后端使用 MockWeComAuth，每次登录创建新用户，无法与 Authing 用户正确绑定。

**方案**: 实现标准的 OAuth 2.0 后端流程：

1. **数据库添加 `authingUserId` 字段**
   ```sql
   ALTER TABLE users ADD COLUMN authingUserId VARCHAR(255) UNIQUE;
   CREATE INDEX users_authing_user_id_idx ON users(authingUserId);
   ```

2. **后端用 code 向 Authing 换取用户信息**
   ```typescript
   // backend/src/auth/authing.ts
   export class AuthingAuthProvider {
     async exchangeCode(code: string): Promise<AuthingUser> {
       // 1. 用 code 换取 access_token
       const tokenResp = await fetch(`${AUTHING_DOMAIN}/oidc/token`, {
         method: 'POST',
         headers: { 'Content-Type': 'application/json' },
         body: JSON.stringify({
           grant_type: 'authorization_code',
           code,
           client_id: AUTHING_APP_ID,
           client_secret: AUTHING_APP_SECRET,
           redirect_uri: AUTHING_REDIRECT_URI,
         }),
       });

       // 2. 用 access_token 获取用户信息
       const { access_token } = await tokenResp.json();
       const userResp = await fetch(`${AUTHING_DOMAIN}/oidc/me`, {
         headers: { Authorization: `Bearer ${access_token}` }
       });

       return await userResp.json(); // { sub, nickname, email, picture }
     }
   }
   ```

3. **使用 `sub` 作为唯一标识符绑定用户**
   ```typescript
   export async function findOrCreateUser(authingUser: AuthingUser) {
     const existing = await db.query.users.findFirst({
       where: eq(users.authingUserId, authingUser.sub),
     });

     if (existing) return existing;

     const [newUser] = await db.insert(users).values({
       authingUserId: authingUser.sub,
       name: authingUser.nickname,
       email: authingUser.email,
       image: authingUser.picture,
       displayName: authingUser.nickname,
     }).returning();

     return newUser;
   }
   ```

**安全优势**: 后端直接从 Authing 服务器获取用户信息，无法被前端篡改。

## 性能优化

### 技能列表计数优化

技能列表页面的 `count(*)` 查询在大数据量时性能较差。

**详情**: [skill-count-counter-cache.md](./skill-count-counter-cache.md)

**问题**: 当前使用 `SELECT count(*)` 获取总数，无法利用索引，大数据量时延迟高。

**方案**: 引入计数器缓存表，通过在状态变更时维护计数器，避免每次列表查询时执行全表 count。
